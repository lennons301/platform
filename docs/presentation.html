Two repos that work together: Platform is the brain (what should the estate look like?) and Interlude is the hands (dispatch AI agents to do the work). Between them, they aim to make a one-person operation feel like a team.

The Problem

Running multiple side projects solo creates three compounding pressures:

Consistency

Without discipline, tech choices drift. One project uses Supabase Auth, another uses Better Auth, a third has no auth strategy. Secrets end up in .env files. Versions fall behind. Every project becomes its own snowflake.

Cost

This can't cost more than just paying for managed services. Every choice must be free-tier-first. The infrastructure to manage projects shouldn't exceed the cost of running them.

Context

AI agents are the force multiplier — but every agent session starts from zero. Without a shared source of truth, you re-explain conventions, stack choices, and project structure every time.

The solution: encode decisions once, enforce them automatically, and give agents the context they need to work autonomously.

Platform — The Brain

The platform repo is the single source of truth for how every project should look. It defines what (standards), which (choices), and how new (versions) — then verifies (conformity checks).

platform/
├── standards/     5 principles (tool-agnostic rules)
├── choices/       6 decision areas (specific tools)
├── versions/      Target runtime & framework versions
├── products/      Registry of every managed project
├── checks/        Shell scripts that validate conformity
├── templates/     Starters for new projects
└── .github/       CI that runs checks on push

Standards

Five tool-agnostic principles. These don't change when you swap a database — they define what must be true regardless of tooling.

Environments

Exactly 3 environments: dev, stg, prd. Isolated. Production credentials never touch dev.

Secrets

Single remote provider as source of truth. Never committed, never hardcoded. Per-environment scoping.

Local Development

All projects run with local Docker services. If your laptop is offline, development still works.

Documentation

Every project maintains a CLAUDE.md as primary technical reference. Updated at the end of each implementation phase.

Testing

Every project has automated tests running against local services. CI runs them on every PR.

Choices

Each decision area has a decision matrix — not a mandate. Projects pick from evaluated options. Reviewed quarterly.

Environment strategy for Neon: dev = local Docker Postgres, stg = Neon ephemeral branch (free), prd = Neon main branch.

Use the postgres driver (postgres.js), NOT @neondatabase/serverless. The websocket driver breaks local Docker Postgres.

Secrets — Doppler (default)

Canonical provider for all projects. Setup: create project in Doppler dashboard, add secrets for dev/stg/prd. Local dev: doppler run -- npm run dev. Vercel: sync prd environment to Vercel env vars.

.env.local is gitignored and ephemeral — a cache, not the source of truth.

Hosting — Vercel Hobby (default)

Shared across the estate: 100 GB bandwidth, 1M invocations, 4 CPU-hours/mo, 6k build-minutes. Non-commercial restriction — upgrade to Pro ($20/mo) if revenue-generating.

Exception: self-hosted Hetzner VPS for projects needing Docker daemon access (Interlude).

ORM — Drizzle vs Supabase Client

Drizzle: Type-safe, pairs with Neon/Better Auth. Supabase Client: Auto-generated types from schema, tied to Supabase.

UI — shadcn/ui + Tailwind (universal)

All frontend projects. Components are source code, not dependencies. Dark mode via className="dark". Use cn() utility for conditional classes.

Intentional Divergences — Interlude

Not every project can follow defaults. The system handles this by requiring documented reasons, not exceptions or workarounds.

# products/interlude.yaml — divergences section

divergences:
  - standard: hosting
    choice: self-hosted-hetzner
    reason: >
      Development platform needs Docker daemon access
      and persistent containers. Serverless hosting is
      architecturally incompatible.

  - standard: secrets
    choice: env-file
    reason: >
      Single VPS deployment. Doppler adds complexity
      with no benefit when there is one deploy target.

Conformity checks see divergences and report them as ✓* (acknowledged), not ✗ (gap).

Interlude — The Hands

What It Does

A self-hosted platform that lets you dispatch development tasks to AI agents from a mobile-friendly web UI, watch them work in real-time, and receive completed work as GitHub pull requests.

Dispatch from anywhere

Mobile-friendly PWA. Create a task, describe what you want, and an agent picks it up.

Real-time streaming

Watch the agent think, write code, and run commands via SSE-streamed output in a chat UI.

Multi-turn conversations

Containers persist between turns. Send follow-up messages. The agent resumes with full context.

Live preview

Agent starts a dev server? It's proxied to a subdomain with auto-TLS. Preview in an iframe or share the URL.

Cost Discipline

A core design principle: this infrastructure should cost less than just paying for hosted services. The platform encodes cost constraints as first-class decisions.

ServiceCostWhat You Get
Vercel HobbyFreeHosting for all product repos (shared limits)
NeonFree10 Postgres databases, 0.5GB each, branching
SupabaseFree2 projects with auth + storage + realtime
DopplerFreeSecrets management for all projects
Hetzner CX22~€4.50/moVPS for Interlude (Docker, persistent containers)
Claude (API/subscription)VariableAgent compute — per-task budgets ($5 default max)
GitHubFreeRepos, Actions CI, Issues

Total fixed cost: ~€4.50/mo (just the VPS). Everything else is free tier or usage-based with hard caps.

Interlude tracks cost per task (Anthropic API spend). Default budget: $5/task, 50 turns max. You see accumulated cost in the chat UI.

Near Term — Phases 3–5

Phase 3: GitHub as the Interface

Label a GitHub issue interlude → webhook fires → task auto-created → agent works → draft PR filed → PR marked ready on completion → issue gets cost/status comments.

This is the bridge to autonomous operation: the trigger moves from a UI button to a GitHub label.

Phase 4: Notification Bot

Slack or Telegram bot for bidirectional messaging. Get notified when tasks complete. Dispatch tasks from chat. "Fix the login bug on lemons" → agent runs.

Phase 5: Production Hardening

Container resource limits (CPU/memory caps), automated SQLite backups, monitoring/alerting, push notifications.

Medium Term — Platform Gets a UI

Today, the platform repo is a collection of YAML and markdown files. The medium-term vision: it becomes a dashboard.

Platform UI — The Control Plane
  • Estate health dashboard: See conformity status across all projects at a glance. Drill into gaps. Click to trigger remediation.
  • Choice browser: Interactive decision matrices. Compare options, see which projects use what, model cost impact of changes.
  • Version tracker: Which projects are behind? One-click upgrade dispatch (agent creates upgrade PR).
  • Task feed: Unified view of all agent work across the estate. Filter by project, status, cost.
  • Project onboarding: Wizard that scaffolds a new project from templates, provisions services (Doppler, Neon, Vercel), and runs conformity checks before a human writes a line.
Phase 6: On-Demand Compute

Cloud provider API for dynamic machine provisioning. Orchestrator decides local VPS vs cloud based on task complexity and queue depth. Auto-teardown after completion.

Phase 7: Multi-Agent Workflows

Agent roles (architect, implementer, reviewer). Task decomposition into subtasks. DAG execution — agent A's output feeds agent B. Parallel branches with automated integration.

Blue Sky — Autonomous Estate Management

The end state: the platform repo evolves from a document into a control plane, and Interlude evolves from a task runner into the execution engine for that control plane.

Self-Healing Conformity

Conformity gap detected → agent automatically creates a branch, fixes the drift, files a PR. No human involvement for routine fixes (missing doppler.yaml, outdated version, missing CLAUDE.md section). Human review for anything structural.

Automated Upgrades

New version of Next.js released → agent upgrades a low-risk project first → validates (tests pass, build succeeds, conformity checks green) → rolls out across the estate in priority order. Version manifest updates automatically.

Self-Provisioning Projects

Create a new product in the platform UI → agent scaffolds from templates, provisions Doppler project, creates Neon database, sets up Vercel project, wires CI/CD, runs conformity checks, and files a PR with the bootstrapped repo. Passes all checks before a human writes a line of code.

Cost-Aware Operations

Monitoring agents watch spend across the estate. Free tier approaching limit? Agent flags it. A project could be cheaper on a different stack? Agent models the impact and proposes a migration plan. Budget exceeded on a task? Hard stop, notify, explain.

Agents Maintaining Agents

Agents upgrade their own tooling — Claude Code versions, container images, CLAUDE.md files refined based on what produces the best results. The platform learns from agent performance data: which prompts, which context, which constraints lead to the best outcomes.
