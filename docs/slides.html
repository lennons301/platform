I've been using Claude (under my subscription) to help build and manage a portfolio of side projects. This is a walkthrough of the system I've put together — part standards framework, part AI agent platform.

The Problem

Consistency

Without discipline, tech choices drift. Every project becomes its own snowflake.

Cost

This can't cost more than just paying for managed services. Free-tier-first, always.

Context

AI agents start from zero every session. Without shared truth, you re-explain everything.

Three pressures that compound over time. Consistency is the obvious one — but cost matters because this is side projects, not a company. And context is the AI-specific one: agents are powerful but amnesiac.

The Solution

Platform — The Brain

Standards, choices, version targets, conformity checks. A single source of truth consumed by both humans and AI agents.

Interlude — The Hands

Dispatch tasks to AI agents from your phone. Watch them work. Get PRs back. Agents inherit platform context automatically.

Two repos that work together. Platform defines what the estate should look like. Interlude gives agents the ability to work on it. Between them, they aim to make a one-person operation feel like a team.

Platform — The Brain

Simple structure. Standards are the "why", choices are the "what", versions are the "how new". Products is the registry. Checks enforce it all. Templates bootstrap new projects.

Standards

Five tool-agnostic principles — these don't change when you swap a database.

Environments

Exactly 3: dev, stg, prd. Isolated. Prod creds never in dev.

Secrets

Single remote provider. Never committed. Per-environment.

Local Development

Docker services. Offline-capable. No remote dependencies.

Documentation

CLAUDE.md in every repo. Updated each phase. Platform context pointer.

Testing

Automated tests. Local services. CI on every PR.

These are the non-negotiables. Tool-agnostic means if we decide to move from Neon to Supabase, the standards still hold. The CLAUDE.md standard is the AI-specific one — it ensures every agent has context.

This is what a fully conformant project looks like. Zero divergences. Every choice is the default. This is the reference implementation — if you're starting a new project, it should look like this.

Intentional Divergences

# products/interlude.yaml
divergences:
  - standard: hosting
    choice: self-hosted-hetzner
    reason: >
      Needs Docker daemon access and persistent
      containers. Serverless is incompatible.

  - standard: secrets
    choice: env-file
    reason: >
      Single VPS. Doppler adds complexity with
      no benefit for one deploy target.

Divergences are ✓* acknowledged, not ✗ gaps. The system handles exceptions with documentation, not rigidity.

This is important — the system isn't rigid. If a project has a good reason to diverge, it documents why and the conformity checks see it as intentional, not broken.
