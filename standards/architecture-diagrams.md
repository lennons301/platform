# Architecture Diagrams Standard

Every project maintains C4 architecture diagrams as PlantUML source files
in `docs/architecture/`.

## Required Diagrams

All projects must include at minimum:

| File | C4 Level | Purpose |
|------|----------|---------|
| `context.puml` | L1 System Context | The system, its users, and external dependencies |
| `containers.puml` | L2 Container | Deployable units (web app, API, database, etc.) |

## Optional Diagrams

Tracked per project in the product YAML's `architecture.diagrams` list:

| File pattern | C4 Type | When to include |
|-------------|---------|-----------------|
| `components-<name>.puml` | L3 Component | Complex subsystems warranting internal breakdown |
| `deployment.puml` | Deployment | Maps containers to infrastructure per environment |
| `dynamic-<name>.puml` | Dynamic | Runtime flows (CI/CD pipelines, key request paths) |

## Format

- Use [C4-PlantUML](https://github.com/plantuml-stdlib/C4-PlantUML) library includes.
- Source `.puml` files are the primary artifact — locally renderable in any IDE with a PlantUML extension.
- Rendered `.svg` files are generated by CI into `docs/architecture/rendered/` and committed back.
- Mark rendered SVGs as binary in `.gitattributes`:
  ```
  docs/architecture/rendered/*.svg binary
  ```

## Maintenance

Diagrams must be updated when architecture changes materially: new containers,
new external systems, changed deployment topology. Not every code change
requires a diagram update.

Three mechanisms help enforce freshness:

1. **Implementation plan convention** — plans that change architecture include
   a diagram update task.
2. **CI staleness detection** — the conformity check warns when
   architecture-relevant files are newer than the diagrams.
3. **PR-level nudge** — a GitHub Action comments on PRs that touch
   architecture-relevant files without also updating diagrams.

## Product YAML

Each product YAML includes an `architecture` section:

```yaml
architecture:
  diagrams:
    - context
    - containers
  staleness_paths:
    - docker-compose.yml
    - vercel.json
```

`context` and `containers` are enforced as minimum regardless of what is listed.
Additional entries declare optional diagrams the project commits to maintaining.

`staleness_paths` lists files whose changes suggest diagrams may need updating.
Defaults are applied if omitted: `docker-compose.yml`, `vercel.json`,
`supabase/`, and common deployment config files.

## Templates

Use `templates/architecture/` from the platform repo as a starting point for
new projects. Skeleton diagrams include placeholders for project name and
description.
